# R3.3 — Visible keyboard focus states

**Status:** planned
**Files:** `src/index.css` (global focus-visible rule), spot fixes in `src/components/Navbar.tsx`
/ `src/routes/home/HeroSection.tsx` only if a control can't take the global rule.

## Spec

**Problem:** Keyboard users get little/no visible focus indicator on the custom anchors and
controls (navbar links, hero "Learn More"/"Join Us", footer social links, project cards,
section anchor links). The dark theme makes the browser default outline nearly invisible.
shadcn primitives (accordion, sheet, navigation-menu) ship their own focus rings, but the
hand-rolled `<a>`/`<button>` elements do not. WCAG 2.4.7 (focus visible) + 2.2 AA, per the new
`design/accessibility.md` (focus: always-visible 2px ring, 2px offset).

**Approach:** Add a single global `:focus-visible` rule (so it only shows for keyboard/AT, not
mouse clicks) in `src/index.css` — which is currently near-empty, making it the right home for
global base styles. Use a visible ring against the dark theme:
```css
:where(a, button, [role="button"], [tabindex]):focus-visible {
  outline: 2px solid #fff;
  outline-offset: 2px;
  border-radius: 2px;
}
```
Using `:where()` keeps specificity 0 so shadcn's own `focus-visible:ring-*` utilities still win
where present (no double ring). White ring chosen (matches the monochrome brand; high contrast
on black) per `design/tokens.md` — recorded in `accessibility.md`.

If any interactive element isn't reachable/!focusable by keyboard, fix that specific element
(e.g. ensure it's an `<a>`/`<button>`), but prefer the global rule.

Decision: white 2px ring + 2px offset, `:where()` for zero specificity. Recorded; no Open Q.

State: none.
