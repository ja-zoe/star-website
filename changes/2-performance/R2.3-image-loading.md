# R2.3 — Image loading discipline

**Status:** planned
**Files:** `src/routes/home/EboardSection.tsx`, `src/components/HoverBlurCards.tsx`,
`src/components/ProjectCard.tsx`, `src/components/Navbar.tsx`, `src/components/Footer.tsx`

## Spec

**Problem:** No `<img>` carries `loading`, `decoding`, or intrinsic `width`/`height`. The
E-Board portraits (8) and sponsor logos (7) are below the fold yet load eagerly, and the lack
of dimensions risks layout shift (CLS) as images arrive.

**Approach:** Add loading hints + explicit dimensions; do NOT change layout (Tailwind classes
keep controlling rendered size — width/height set the intrinsic ratio for CLS only).
1. **EboardSection portraits:** `loading="lazy"` `decoding="async"` `width={640} height={640}`
   (matches the R2.1 webp; the circle/scale classes still control display).
2. **Sponsor logos (HoverBlurCards):** `loading="lazy"` `decoding="async"`. These are small
   PNGs of varying aspect; omit fixed width/height to avoid distortion (the `w-32` class sizes
   them) — loading/decoding hints are the win here.
3. **ProjectCard icon:** `loading="lazy"` `decoding="async"` `width={66} height={66}`
   (below the hero, fixed 66px).
4. **Navbar logo:** keep eager (it's above the fold / LCP-adjacent) but add `width`/`height`
   to reserve space and `fetchPriority="high"`. Footer icons: `loading="lazy"` `width`/`height`.

Decision: the navbar logo stays eager + high priority (it's first paint); everything else goes
lazy. Recorded.

State: none.
