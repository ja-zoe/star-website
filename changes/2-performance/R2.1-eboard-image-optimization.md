# R2.1 — Eboard image optimization

**Status:** planned
**Files:** `package.json` (sharp devDep + script), `scripts/optimize-images.mjs` (new),
`public/eboard/*.webp` (new, generated), `src/routes/home/EboardSection.tsx` (refs)

## Spec

**Problem:** The 8 e-board portraits are 5.2–6.7 MB JPEGs (~47 MB total) and are rendered in
240px-diameter circles (`w-60 h-60`, `scale-125`). On the home page every visitor downloads
~47 MB of imagery for thumbnails — the single largest performance defect on the site
(measured). At 2× DPR the displayed image never exceeds ~600 px.

**Approach (build-time, static-safe — produces committed static files Apache serves as-is):**
1. Add `sharp` as a **devDependency** (build-time only; never imported at runtime, so the
   deployed bundle stays a pure static SPA — does not violate the static-host invariant).
2. `scripts/optimize-images.mjs` (Node, ESM): for each source portrait in `public/` (julian,
   sandhya, shilpi, saarim, praneeth, aayushi, nila, kanika `.jpg`), produce
   `public/eboard/<name>.webp` resized to **640×640**, `fit: cover`, center-ish (the existing
   per-member `objectPosition` tweaks stay in CSS), quality ~78. Idempotent (skip if up to
   date is optional; simplest: always overwrite). Log original vs new byte sizes.
   Add `"optimize:images": "node scripts/optimize-images.mjs"` to package.json scripts.
   Note in the script header that it is a one-shot dev tool, NOT wired into `pnpm build`
   (keeps CI/build sharp-free and fast; outputs are committed).
3. Run it once, commit the generated `public/eboard/*.webp`.
4. `EboardSection.tsx`: change the 8 imports from `/julian.jpg` → `/eboard/julian.webp`, etc.
   Keep the original JPGs in `public/` (do not delete source images — they're the regeneration
   source and may be reused; only the *referenced* asset changes).

Decision: 640×640 WebP q78 chosen — comfortably covers the 240px circle at up to ~2.6× DPR
while collapsing each file from ~6 MB to roughly 40–90 KB. Recorded; no Open Question.

State: none.
