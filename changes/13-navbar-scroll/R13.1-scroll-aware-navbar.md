# R13.1 — Scroll-aware navbar underlay

**Status:** planned
**Files:** `src/components/Navbar.tsx`

## Spec

**Problem:** The global `Navbar` is `fixed top-0` and fully transparent, so its white text/logo
clashes with whatever scrolls under it (big outline headings, e-board photos, the colorful hero
waves, etc.) depending on scroll position — a legibility problem. The user likes the transparent
look but wants a solid underlay when content clashes.

**Approach (design/components.md › Navigation bar):** make the navbar scroll-aware.
- Add `const [scrolled, setScrolled] = useState(false)` to `Navbar`. In a `useEffect`, attach a
  **passive, rAF-throttled** `scroll` listener on `window`: `setScrolled(window.scrollY > 24)`.
  Call once on mount (for refresh-at-scroll), and clean up the listener + any pending rAF on
  unmount. Guard `typeof window` (client-only SPA, but defensive).
- The root `<header>` gets a base `transition-colors duration-300` and, when `scrolled`,
  `bg-black/80 backdrop-blur-md border-b border-white/10` (use `cn()` from `../lib/utils`).
  At the top: no background (transparent, current look). Scrolled: solid blurred underlay so nav
  text is always legible over page content.
- Reduced-motion: the global `@media (prefers-reduced-motion: reduce)` (set 1 R1.4) already zeroes
  transition-duration, so the underlay snaps instead of fading — no extra work.

No new deps. Threshold 24px (just past the top). Mobile Sheet menu unaffected (it has its own bg).

Alternatives considered + rejected (for the record):
- **Per-pixel clash detection** (sample rendered pixels under the bar each frame): fragile and
  expensive, especially over the WebGL/canvas star + wave layers. Over-engineered.
- **Always-on gradient scrim** (a permanent `from-black/70 to-transparent` strip): simplest, zero
  JS, but slightly dims the top even over the clean hero; the scroll-state underlay better honors
  "I like the transparent look" at the top.
