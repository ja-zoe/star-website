# R5.1 — .htaccess caching + compression

**Status:** planned
**Files:** `public/.htaccess`

## Spec

**Problem:** `public/.htaccess` (the file Apache serves the SPA from) sets up SPA fallback +
ErrorDocuments but adds **no cache or compression headers**. Vite emits content-hashed assets
(`assets/index-HASH.js`, `eboard/*.webp`, etc.) that are safe to cache for a year, while
`index.html` must NOT be cached (so new deploys are picked up). Today everything is served with
default (often weak/no) caching and uncompressed — wasted bandwidth + slow repeat visits on the
bare Apache host.

**Approach:** Append, to the existing `.htaccess`, three `<IfModule>`-guarded blocks (guards =
if a module isn't enabled on the host, Apache skips the block instead of 500-ing):
1. **mod_deflate** — gzip-compress text types: html, css, js, svg, json, xml, plain.
2. **mod_headers + mod_expires** — `Cache-Control`:
   - `/assets/*` and other content-hashed files (js, css, woff2, webp, png, jpg, gif, svg):
     `public, max-age=31536000, immutable` (1 year).
   - `index.html` (and `.html`): `no-cache, must-revalidate` so the entry document always
     revalidates and deploys propagate.
   Use `mod_expires` `ExpiresByType` for the long-lived types and a `<FilesMatch "\.html$">`
   `Header set Cache-Control` for the no-cache HTML, so the rules don't fight.
3. Keep all existing rules (RewriteEngine SPA fallback + ErrorDocuments) untouched and first.

Decision: hashed-asset rule keys off file extension (Vite hashes the names, so even same-ext
files bust cache on change); HTML explicitly no-cache. immutable used for the hashed assets.
Recorded; no Open Question.

State: none (server config; still static — Apache directives only, no runtime).
